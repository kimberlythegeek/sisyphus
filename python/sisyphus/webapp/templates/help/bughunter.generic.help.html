Overview

The purpose of bughunter is to help find bugs by collecting extensive meta data associated with crash reports, assertions, and valgrind data. This user interface enables a user to browse this meta data by connecting datasets together. You can find detailed info about the application architecture here. The application presents data in data views. Data views can be presented in multiple panes within a single browser page or displayed in a separate pages depending on what works best for a user's display. Data views can be connected to each other when they share a data type or identifier allowing a user to explore deep relationships in the data.

View Navigation

Each data view has a navigation menu displayed in the top left panel. When a new navigation position is chosen the associated view will initialize accordingling. If a data collection is selected the browser page will be cleared of all open views and replaced with the view collection that was selected.

Control Panel

The control panel is composed of a form with fields that pertain to the view displayed. The form values can be modified and the view reloaded by selecting the load button at the bottom of the control panel. The control panel form values are only applied to the associated view, with the exception of the date range.

The date range is passed on as signal data to any child view, so you can control the date range of a set of child views by making a single modification in the parent's control panel.

To look for similiarity in a particular field, first edit the field to contain the target text then insert a '%' at the beginning, end, or both beginning and end of the field data.

Parent/Child Relationships

Data views in bughunter have a parent/child relationship that is indicated in the top panel of each view. A view's parent can send signals to its child views when the user selects links or graphical representations of data that can send signals. The parent/child relationship of any view is identified by a data view number "View:Some Number" that refers to the data view seleted and its parent "Parent:Some Page Name, Some View Name". Anytime a new view is opened from an existing view, that view becomes the parent. Parents can send signals to child views that exist on different browser tabs or pages and there is no limit to the number of child views a parent can have.

Visualization

View data can be visualized in a tabular or graphical form. Different visualizations of the data from a view can be accessed using the Visualization menu, some views will only have a tabular presentation.

Button Controls

Each view has a set of button controls displayed in the top panel. The following is a description of each button from left to right.

Decrease View Panel Size: Decrease the display size of a view's data table. This may or may not apply to a graphical representation.

Increase View Panel Size: Increase the display size of a view's data table. This may or may not apply to a graphical representation.

Open New View: This button opens a modal window that presents the user with options for opening new child views. The child views opened will be parented to the view from which they were opened.

Signals

Signals can be sent from a parent view to its children by selecting links displayed in tabular data or graphical representations of data that can send signals. When a signal is sent, it's displayed in the parent view in the middle panel and prefixed with "Sent:", when a signal is received by a child view it's displayed in the parent view in the middle panel and prefixed with "Received:". A date range is always displayed in the signal display panel and is prefixed with "Range:". The date range can be set by opening the control panel and modifying the date range displayed.

Site Crashes Overview

Site crash data is organized into three different types of data representations: Crashes, URL summary, and Related Crashes. These datasets enable a user to explore crash data within a time range.

Site Crashes

signature: When a crash occurs, a minidump file is created by breakpad. The minidump file is uploaded to the database server and the path to the raw file is saved in the minidump field. This minidump file is processed by a stack walking program that creates a crash report. The crash signature is created by taking the stack frames from the stack in the crash report and munging it to produce a signature which can be used to group similar crashes. Extraneous addresses, numbers, and non-alpha characters have been removed to improve reproducibility.

fatal message: The warning/error message found in the log.

Total Count: The total number of times this particular signature has occured.

Platform: The platform field is split into the following pieces of data: source tree, operating system, version, architecture, and number of occurances. Any number of platforms can be associated with a given crash signature.

Site Crash URL Summary

url: The url associated with the crash report.

Total Count: The total number of times this url has caused a crash in the database.

Platform: The platform field is split into the following pieces of data: source tree, operating system, version, architecture, and number of occurences. Any number of platforms can be associated with a given crash signature.
