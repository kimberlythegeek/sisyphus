Overview

Bughunter consists of a backend where worker machines perform tests and submit the results to a database and a frontend user interface where the results can be viewed in a web browser and used to file bug reports. The purpose of bughunter is to help users find and file bugs on crashes, assertions and valgrind warnings.

Bughunter performs two types of tests and organizes the results accordingly:

Site Tests

Site Tests involve testing the behaviour of the browser when loading web pages from various web sites. Currently, Site Tests consist of attempting to reproduce crashes by loading the urls reported to Socorro.

UnitTest Tests

UnitTest Tests involve running the UnitTests on Linux and Mac OS X under Valgrind's memcheck in order to discover any memory related problems as well as to record assertions encountered during the Unit Tests.

Bughunter's user interface enables a user to browse this data by connecting datasets together. You can find detailed info about the application architecture here. The application presents data in data views. Data views can be presented in multiple panes within a single browser page or displayed in separate pages depending on what works best for a user's display. Data views can be connected to each other when they share a data type or identifier allowing a user to explore deep relationships in the data.

View Navigation

Each data view has a navigation menu displayed in the top left panel. When a new navigation position is chosen the associated view will initialize accordingly. If a data collection is selected the browser page will be cleared of all open views and replaced with the view collection that was selected.

Control Panel

The control panel is composed of a form with fields that pertain to the view displayed. The form values can be modified and the view reloaded by selecting the load button at the bottom of the control panel. The control panel form values are only applied to the associated view, with the exception of the date range.

The date range is passed on as signal data to any child view, so you can control the date range of a set of child views by making a single modification in the parent's control panel.

To look for similiarity in a particular field, first edit the field to contain the target text then insert a '%' at the beginning, end, or both beginning and end of the field data.

Parent/Child Relationships

Data views in bughunter have a parent/child relationship that is indicated in the top panel of each view. A view's parent can send signals to its child views when the user selects links or graphical representations of data that can send signals. The parent/child relationship of any view is identified by a data view number "View:Some Number" that refers to the data view selected and its parent "Parent:Some Page Name, Some View Name". Anytime a new view is opened from an existing view, that view becomes the parent. Parents can send signals to child views that exist on different browser tabs or pages and there is no limit to the number of child views a parent can have.

Visualization

View data can be visualized in tabular or graphical form. Different visualizations of the data from a view can be accessed using the Visualization menu, some views will only have a tabular presentation.

Button Controls

Each view has a set of button controls displayed in the top panel. The following is a description of each button from left to right.

Decrease View Panel Size: Decrease the display size of a view's data table. This may or may not apply to a graphical representation.

Increase View Panel Size: Increase the display size of a view's data table. This may or may not apply to a graphical representation.

Open New View: This button opens a modal window that presents the user with options for opening new child views. The child views opened will be parented to the view from which they were opened.

Help: Clearly you understand this button...

Refresh: Reload the data from its original source in the view.

Move To New Window: Move the view data to a new browser tab or window.

Close: Close the view.

Signals

Signals can be sent from a parent view to its children by selecting links displayed in tabular data or graphical representations of data that can send signals. When a signal is sent, it's displayed in the parent view in the middle panel and prefixed with "Sent:", when a signal is received by a child view it's displayed in the parent view in the middle panel and prefixed with "Received:". A date range is always displayed in the signal display panel and is prefixed with "Range:". The date range can be set by opening the control panel and modifying the date range displayed.

Table Filters

Directly above each tabular display of data is a set of controls that operate on the table. The controls are described below from left to right.

Show Entries: This option menu can be used to changed the number of rows displayed on a single page. When the number of rows exceeds the page display limit a set of paging links are displayed in the lower right of the table.

Column Filter: This option menu can be used to change which columns are displayed. When opened, all of the columns available in a table are displayed along with a checkbox that can be used to toggle the display of a column.

Search Filter: The search filter will filter all of the data contained within a data table by any string or regular expression entered into the input box.

Context Menu

A context menu is available for cells in the table that contain links. Options are presented above the standard browser options that are specific to bughunter. Not all of the options described are available for all data types.

Search Bugzilla: This option provides a relevent set of bugzilla field searches. If no data in the cell is selected the search will include all of the contents of the cell. The data used in the query can be limited by selecting a subset of the cell data.

Search crash-stats: This option executes a search on the crash-stats database. If no data in the cell is selected the search will include all of the contents of the cell. The data used in the query can be limited by selecting a subset of the cell data.

Select Cell Data: When selected this option will automatically select all of the text in a data cell for convenient copy/paste operations.

Copy Cell Data: When selected this option will copy the contents of a given cell to the clipboard. This feature requires the user to enable the ability in firefox. You can allow access by entering 'about:config' in the location bar in the browser and setting 'signed.applets.codebase_principal_support=true' or installing the AllowClipboard addon and enabling clipboard access for bughunter.

Open URL In Window: This is only applicable to links that have a URL for content. When selected, this will open the url in a new browser window. USE CAUTION: These urls may contain very disturbing content and might crash your browser: shut the door, cover your eyes, and prepare for the worst...

Site Test Data

Site test data consists of the Crashes, Assertions, and potentially Valgrind warnings generated during Site Testing.

Site Crashes Overview

Site crash data is organized into three different types of data representations: Crashes, URL summary, and Related Crashes. These datasets enable a user to explore crash data within a time range.

Site Crashes

signature: When a crash occurs, a minidump file is created by breakpad. The minidump file is uploaded to the database server and the path to the raw file is saved in the minidump field. This minidump file is processed by a stack walking program that creates a crash report which is also uploaded to the crashreport field. The crash signature is created by taking the stack frames from the stack in the crash report and munging it to produce a signature which can be used to group similar crashes. Extraneous addresses, numbers, and non-alpha characters have been removed to improve reproducibility.

fatal message: The warning/error message generated by a fatal assertion such as an Assertion failure or ABORT.

Crash Table Overview

Crash tables are a collection of one-off tables that hold useful meta data that does not fit into the site/unit categories. Currently, the only crash table is Socorro Record which provides a way to connect the bughunter Crash data to the original crash data source in the socorro database.

Socorro Record

The Socorro Record contains the data from the user's original Socorro crash report. With the exception of the socorro_id, each of these fields corresponds to a similar field in the crash report available from crash-stats.mozilla.com.

socorro_id: The socorro id is the primary key for the Socorro Record table. It is also a foreign key in the Site Test Run table. socorro_id signals can be sent to and from the Socorro Record view and the Site Test Run view. In this way you can look up the Site Test Runs associated with a Socorro Record for look up the specific Socorro Record assocated with a Site Test Run.

signature: The socorro crash signature which is distinct from the bughunter signature.

url: The url associated with the crash report.

uuid: The crash report id of the original Socorro crash report.

client_crash_date: The date of the crash.

date_processed: The Date Processed from the original Socorro report.

last_crash: The Last Crash Date

product: The name of the software product which generated the crash report.

build: The Build ID of the crashing product.

branch: The source branch.

os_name: The name of the operating system associated with the crash.

os_full_version: The full version of the operating system associated with the crash.

os_version: The abbreviated version of the operating system associated with the crash.

cpu_info: Description of the cpu.

cpu_name: Name of the cpu.

address: The Crash Address from the Socorro crash report.

bug_list: A comma delimited list of the bug numbers which are associated with this crash report in Socorro.

user_comments: The user comments associated with the crash.

reason: The Crash Reason from the Socorro crash report.
